Standalone add-on for KDNA Events that generates branded PDF tickets with Code 128 barcodes and attaches them to booking confirmation emails.

== Description ==

**This is a standalone add-on. It requires the core KDNA Events plugin (version 1.2.0 or higher) to be installed and active.** The add-on does not modify core in any way. It extends core through the three extension hooks added in v1.1 Brief A (`kdna_events_email_attachments`, `kdna_events_after_success_ticket`, `kdna_events_after_my_ticket`) and shares the Dompdf library shipped with core v1.2+ rather than vendoring its own copy, which keeps this add-on under 1 MB.

Every ticket generated by core gets a branded PDF with a Code 128 barcode for scanner-based entry. The PDF is attached to the customer booking confirmation email, and download buttons appear on the Success Tickets and My Tickets widgets so buyers can fetch a copy at any time.

= What this add-on does =

* Generates branded PDF tickets, one per page, at A4 or Letter, portrait or landscape.
* Attaches the PDFs to the booking confirmation email (not the admin notification).
* Adds a 'Download Ticket PDF' button below each ticket on the Success page and the My Tickets dashboard.
* Exposes a REST endpoint for authenticated PDF downloads with optional signed-token access for guest buyers.
* Provides a 'PDF Tickets' settings page under the core Events menu with full brand, colour, typography, barcode and layout controls.
* Inherits design tokens from core's Email Design tab by default, with per-token override toggles.
* Code 128 barcode encoding the ticket code, human-readable text printed below for staff verification.

= What this add-on does NOT do =

* Does not modify core KDNA Events files.
* Does not generate QR codes (Code 128 only in v1.0; QR is a future feature).
* Does not provide per-event PDF customisation; the template is global.
* Does not run independently of core.

== Installation ==

1. Install and activate the core KDNA Events plugin first (version 1.1.0 or higher).
2. Upload this add-on to `/wp-content/plugins/kdna-events-pdf-tickets/` or install via the Plugins screen.
3. Run `composer install --no-dev` inside the add-on folder so `vendor/` is populated, unless you installed from a release zip that already ships `vendor/`.
4. Activate the plugin. A 'PDF Tickets' submenu appears under the Events menu.
5. Configure your brand tokens (logo, colours, typography) or leave every 'Inherit from core Email Design' toggle on to share branding with the customer email.

== Dompdf and sharing ==

Core KDNA Events 1.2.0 ships Dompdf 2.x for tax invoice generation. This add-on does NOT vendor its own copy of Dompdf, it relies on the core instance. That keeps the add-on's zip under 1 MB. If you deactivate core or strip its `vendor/` directory, the add-on detects Dompdf is missing and shows an admin notice explaining why. Barcode generation uses `picqer/php-barcode-generator`, vendored inside this add-on.

== Frequently Asked Questions ==

= How do signed download tokens work? =

Each 'Download Ticket PDF' link carries a `?t=...` query string holding an HMAC-signed token that expires after 24 hours. The REST endpoint validates the token against the ticket code so guest buyers can download their own ticket without logging in, while strangers cannot enumerate other people's tickets.

= Can I use a custom font? =

Dompdf supports TTF fonts if they are registered through its font metric cache. The add-on's font selector lists the three core fonts (Helvetica, Times, Courier) that Dompdf ships by default. Custom font support is flagged as a future enhancement; for now, stick to a web-safe fallback stack.

= How do I switch to QR codes later? =

QR is reserved for a future update that will ship alongside a wallet pass add-on. The stored ticket code does not change, only the rendered graphic. Any ticket printed with Code 128 today will still scan at the door when QR arrives, because most mobile scanner apps read both formats.

= Dompdf is throwing memory warnings on my host =

Dompdf is memory-hungry on complex layouts. Increase `WP_MEMORY_LIMIT` to 128M and ensure your host allows `isRemoteEnabled` so remote images (logos, event banners) can render. The add-on sets both defaults.
